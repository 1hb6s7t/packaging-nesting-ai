# Batch Layout Architecture

This document records the current enterprise batch layout flow and the persistence contract that connects batch artwork intake, pattern planning, Top3 selection, approval, and export.

## Flow

1. `BatchArtworkService` creates a `batch_upload`, stores `batch_artwork_item` rows, runs preflight, parses SVG/DXF geometry, and records feature/classification data.
2. `BatchLayoutService.create_job()` stores the `batch_layout_job`, parent sheet, and cut variants.
3. `CompatibilityGroupingService` groups parsed items by material, thickness, print method, spot color, due date, category, and customer.
4. `PatternPlanner` creates candidate `ProductionPattern` rows for each group and cut variant.
5. `MultiSolverOrchestrator` is invoked for candidate-pool evidence so Top3 plans cannot rely only on a single heuristic output.
6. `TopKGlobalPlanSelector` selects highest-utilization, balanced-risk, and fastest-production plans.
7. `ProductionPlanBuilder` writes validator summaries, audit manifests, quantity reports, diversity score, and export gate flags.
8. Approval and export remain separate workflows. AI tools cannot approve or export production artifacts.

## Placement Contract

The placement artifact is generated by deterministic system code, not by AI. Each artifact includes:

- schema version and artifact type
- job, group, pattern, and cut-variant identifiers
- sheet dimensions and cut-variant metadata
- deterministic solver identity and input SHA-256
- rendered sheet-template placements
- `complete_item_coverage`, `omitted_item_count`, and truncation flags
- a pointer to `validator_report.quantity_summary` for complete per-item quantity accounting

Large compatibility groups keep the artifact bounded by rendering representative templates while keeping full quantity accounting in validator JSON. This keeps 1500/20000 release gates practical and makes any rendering truncation explicit.

## Remaining Production Boundary

The current placement artifact is deterministic and auditable, but it is still not a configured external PackingSolver/Sparrow binary acceptance artifact. Native PDF die-line geometry and conversion-supplier acceptance also remain separate go-live evidence requirements.
